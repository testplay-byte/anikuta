# Video Player

The aniyomi video player is an MPV-backed (not ExoPlayer/Media3) single-Activity player that loads anime episodes by resolving a `Hoster` → `Video` stream from an `AnimeSource` extension, then drives `libmpv` through JNI while a Compose overlay renders all controls.

## Where it lives

Code roots (paths relative to repo root, all under `REFERENCE/`):

- `app/src/main/java/eu/kanade/tachiyomi/ui/player/` — Activity, ViewModel, MPV view wrapper, observer, PiP, external intents, enums, utils, and sub-packages:
  - `controls/` — Compose overlay (root `PlayerControls`, `GestureHandler`, four corner controls, `PlayerSheets`, `PlayerPanels`, `PlayerDialogs`) and `controls/components/` (seek bar, vertical sliders, sheets, panels, dialogs).
  - `loader/` — `EpisodeLoader.kt`, `HosterLoader.kt` (resolve hosters/videos from the source).
  - `settings/` — five `*Preferences` classes (`PlayerPreferences`, `GesturePreferences`, `DecoderPreferences`, `AudioPreferences`, `SubtitlePreferences`, `AdvancedPlayerPreferences`).
  - `utils/` — `AniSkipApi.kt`, `TrackSelect.kt`, `ChapterUtils.kt`.
- `app/src/main/java/eu/kanade/presentation/player/components/` — shared Compose building blocks (`PlayerSheet`, `ExpandableCard`, `OvalBox`, `TintedSliderItem`, `SliderItem`, `RepeatingIconButton`, `OutlinedNumericChooser`, `ExposedTextDropDownMenu`, `SwitchPreference`). Used by both the in-player overlay and the settings screens.
- `app/src/main/java/eu/kanade/presentation/more/settings/screen/player/` — player settings screens (main, player, subtitle, audio, decoder, gestures, advanced, custom-button, editor).
- `app/src/main/res/layout/player_layout.xml` — the Activity's only XML layout: a `ConstraintLayout` containing one `AniyomiMPVView` (the video surface) + one `ComposeView` (all controls).
- `app/src/main/AndroidManifest.xml` — `PlayerActivity` registered at line 196 with `singleTask`, `supportsPictureInPicture=true`, `resizeableActivity=true`, `autoRemoveFromRecents=true`, and `configChanges` for orientation/screenSize/uiMode/keyboardHidden (the Activity handles config changes itself).

## What it does

- Plays a single anime episode end-to-end: given an `animeId` + `episodeId` (and optional pre-resolved `hostList`/`hostIndex`/`vidIndex`), resolve the stream URL via the source, hand it to MPV, render controls, save progress every second, mark seen, advance trackers, optionally auto-play next episode, optionally delete after seen.
- Renders the entire player chrome (top/bottom/left/right controls, seek bar, sheets, panels, dialogs, gesture overlay, double-tap-to-seek ovals, brightness/volume sliders, skip-intro button, screenshot sheet) in Jetpack Compose on top of the MPV `SurfaceView`.
- Supports: hardware/software decoder switching, GPU/GPU-next vo, debanding, YUV420P, video filters (brightness/saturation/contrast/gamma/hue), audio/subtitle track selection + external track loading, dual subtitles (`secondary-sid`), subtitle styling (font, size, color, border, shadow, justification, ASS override), audio delay, pitch correction, volume boost beyond system max, playback speed (0.25–4.0), sleep timer, video aspect (crop/fit/stretch), screen orientation per video aspect, Picture-in-Picture with media controls, MediaSession, AniSkip skip-times (op/ed/recap/mixed-op) with auto-skip or Netflix-style countdown, custom Lua buttons, mpv.conf/input.conf/user-scripts/user-shaders/fonts, screenshots (save/share/set as cover/background/thumbnail), episode list dialog, chapter navigation, autoplay next episode, download-ahead-while-watching, incognito mode, external player fallback.

## How it works

### Player library: MPV (not ExoPlayer / Media3)

Confirmed in `app/build.gradle.kts` line 304: `implementation(aniyomilibs.aniyomi.mpv)`. Catalog `gradle/aniyomi.versions.toml` line 11: `aniyomi-mpv = { module = "com.github.aniyomiorg:aniyomi-mpv-lib", version.ref = "aniyomi-mpv-lib" }` with `aniyomi-mpv-lib = "1.18.n"`. The native lib is `libmpv.so` (see `packaging.jniLibs.keepDebugSymbols` list at line 117). Java/Kotlin bindings live in package `is.xyz.mpv` — `BaseMPVView` (SurfaceView subclass that owns the MPV instance), `MPVLib` (static JNI bridge: `command(Array<String>)`, `setPropertyString/Int/Boolean/Double`, `getPropertyString/Int/Boolean/Double`, `observeProperty`, `setOptionString`, `addObserver`, `addLogObserver`, event id constants, `mpvFormat` enum), `KeyMapping` (Android keycode → mpv keyname table). The view subclass `AniyomiMPVView` is what `player_layout.xml` instantiates.

Additional player-adjacent libs (same build file): `aniyomilibs.ffmpeg.kit` (`com.github.jmir1:ffmpeg-kit:1.18` — used by some sources via `Video.ffmpegStreamArgs`/`ffmpegVideoArgs`, TODO: not invoked directly from the player package), `aniyomilibs.arthenica.smartexceptions` (FFmpeg ABI detection helper), `aniyomilibs.seeker` (`io.github.2307vivek:seeker:1.2.2` — the seek bar with chapter-tick support), `aniyomilibs.truetypeparser` (`io.github.yubyf:truetypeparser-light:2.1.4` — used by the subtitle font picker to list font names from TTF files), `aniyomilibs.mediasession` (`androidx.media:media:1.7.0` — `MediaSession` for lock-screen / Bluetooth controls).

### Playback lifecycle: user taps episode → video plays

1. **Launch.** Anywhere in the app, `MainActivity.startPlayerActivity(context, animeId, episodeId, extPlayer=false, video=null, hosterIndex=-1, videoIndex=-1, hosterList=null)` (line 591) is called. With `extPlayer=false`, it calls `PlayerActivity.newIntent(context, animeId, episodeId, hosterList, hosterIndex, vidIndex)` and `startActivity`. `newIntent` puts `animeId`, `episodeId`, optional serialized `hostList` (JSON of `SerializableHoster` list), optional `hostIndex`/`vidIndex`, and `FLAG_ACTIVITY_CLEAR_TOP`. Callers: `AnimeScreenModel`, `AnimeLibraryTab`, `AnimeHistoryTab`, `AnimeUpdatesTab`, `DeepLinkAnimeScreen`, `NotificationReceiver` (new-episode notification).
2. **Activity create.** `PlayerActivity.onCreate` (line 222): `enableEdgeToEdge()`, `setContentView(binding.root)`, then `setupPlayerMPV()` — writes `mpv.conf`/`input.conf` from `AdvancedPlayerPreferences` into `filesDir/mpv/`, copies user scripts/shaders/script-opts from `StorageManager` directories (gated by `mpvUserFiles`), copies the bundled `aniyomi.lua` bridge + `subfont.ttf` + `cacert.pem` assets, copies user fonts, then calls `player.initialize(configDir, cacheDir, logLvl)` and registers `PlayerObserver` as log + event observer. Then `setupPlayerAudio()` (AudioFocus request), `setupMediaSession()` (MediaSession with play/pause/skip callbacks routed to `SingleActionGesture` prefs), `setupPlayerOrientation()`. Sets `binding.controls.setContent { TachiyomiTheme { PlayerControls(viewModel, onBackPress, ...) } }`. Finally `onNewIntent(this.intent)`.
3. **Init episode.** `PlayerActivity.onNewIntent` (line 173) saves the current episode's progress, flips `isLoadingEpisode`/`isLoadingHosters`, then `viewModel.init(animeId, episodeId, hostList, hostIndex, vidIndex)`. `PlayerViewModel.init` (line 1175): if `needsInit` → `getAnime.await(animeId)`, wait for `sourceManager.isInitialized`, set `episodeId`, `checkTrackers(anime)`, `updateEpisodeList(initEpisodeList(anime))` (sorts via `getEpisodeSort`, filters downloaded-only, maps to db `Episode`), find the episode in the playlist, `sourceManager.getOrStub(anime.source)`, set `_currentEpisode`/`_currentSource`, write `user-data/current-anime/{anime-title,intro-length,category}` MPV properties, then resolve hosters: if `hostList` non-blank → deserialize it; else `EpisodeLoader.getHosters(episode, anime, source)` (downloaded → build from download manager; HTTP → `source.getHosterList` or legacy `source.getVideoList`; local → file URI). Returns `InitResult(hosterList, videoIndex=qualityIndex, position=episodePosition)`.
4. **Load hosters + pick video.** `PlayerActivity.onNewIntent` then calls `viewModel.loadHosters(source, hosterList, hosterIndex, vidIndex)` (line 1289). This cancels any prior `getHosterVideoLinksJob`, sets `_hosterState` per-hoster (`Idle` if `hoster.lazy`, `Loading` if `videoList==null`, `Ready` otherwise), then `coroutineScope { hosterList.mapIndexed { async { EpisodeLoader.loadHosterVideos(source, hoster) } }.awaitAll() }`. For the hoster at the requested `hosterIndex`, calls `loadVideo(source, video, hosterIndex, videoIndex)`. If no index given (`-1`), picks the first `preferred` video across all hosters, else `HosterLoader.selectBestVideo` picks first non-empty. `loadVideo` (line 1377) calls `HosterLoader.getResolvedVideo(source, video)` (which calls `source.resolveVideo(video)` for `AnimeHttpSource` + `!video.initialized`), updates `HosterState` to `READY`/`ERROR`, sets `_currentVideo`, persists `qualityIndex`, then `activity.setVideo(resolvedVideo)`.
5. **Hand to MPV.** `PlayerActivity.setVideo(video, position)` (line 1051): `setHttpOptions(video)` (writes `http-header-fields` from `video.headers ?: source.headers`), computes resume position (`episode.last_second_seen` unless seen & `!preserveWatchingPosition`, else 0), `MPVLib.command("set", "start", "${resumePos/1000F}")`, joins `video.mpvArgs` into `"k=v,k=v"` string, then `MPVLib.command("loadfile", parseVideoUrl(video.videoUrl), "replace", "0", videoOptions)`. `parseVideoUrl` resolves file/content/data/`http(s)`/`fd://` URIs via `Uri.resolveUri`.
6. **File loaded.** MPV fires `MPV_EVENT_FILE_LOADED` → `PlayerActivity.event(eventId)` → `fileLoaded()` (line 1184): `setMpvOptions()` (applies `Video.MPV_ARGS_TAG` from metadata for downloaded videos), `setMpvMediaTitle()` (writes `force-media-title` + `user-data/current-anime/episode-title`), `setupPlayerOrientation()`, `setupChapters()` (merges `video.timestamps` into chapters), `setupTracks()` (issues `audio-add`/`sub-add` MPV commands for each external `audioTracks`/`subtitleTracks` on the `Video`), then AniSkip lookup if enabled. MPV fires `MPV_EVENT_PLAYBACK_RESTART` → `player.isExiting = false`.

## Anime vs manga

The player is anime-specific. It plays videos via MPV and has no coupling to the manga reader's page-rendering pipeline. There is no shared base class with `ReaderActivity`/`ReaderViewModel`.

Two minor manga-coupling points exist (both utility-level, not architectural):

1. `PlayerViewModel.kt` line 61 imports `eu.kanade.tachiyomi.data.database.models.manga.isRecognizedNumber` (a top-level `Episode.isRecognizedNumber` extension fn reused for the "mark duplicate-numbered episodes seen" feature in `updateEpisodeProgressOnComplete`). This is a shared DB-model utility, not a reader dependency.
2. `PlayerViewModel.kt` line 80 imports `eu.kanade.tachiyomi.ui.reader.SaveImageNotifier` and uses it (line 1734) for screenshot save/share notifications. `SaveImageNotifier` lives in the manga reader package but is a generic image-save notification helper — it does not depend on the reader Activity/ViewModel.

Both could be trivially moved to a shared `util/` package to fully decouple the player from the reader. There is no other manga/reader coupling in the player package.

Relationship to the reader (manga):

- Separate Activities: `PlayerActivity` (anime) vs `ReaderActivity` (manga). Both are `singleTask` and registered in the manifest. No shared base Activity.
- Separate ViewModels: `PlayerViewModel` vs `ReaderViewModel` (in `ui/reader/`). No shared base ViewModel.
- The reader has its own settings (`ReaderPreferences`, `reader/settings/*` Compose pages) and its own Compose overlay (`presentation/reader/`); the player has its own (`PlayerPreferences` etc., `presentation/player/`, `presentation/more/settings/screen/player/`). They share only the `presentation/components/` primitives (AppBar, sheets, dialogs) used app-wide.
- Both feed into the same history/progress SQLDelight tables (anime vs manga are separate tables: `anime_history` vs `history`, `episodes` vs `chapters`).

For an anime-first build: the player package + its loaders + `:source-api` anime models + `AnimeDownloadManager` + anime domain/data interactors form a self-contained slice. The manga reader and its preferences can be dropped entirely; only `SaveImageNotifier` and the `isRecognizedNumber` extension would need to be relocated (or re-implemented) to fully sever the dependency.

## Notes for our build (anime-first)

The player is the core of an anime-first app. Recommendations:

- **Reuse the entire `ui/player/` package as-is** (Activity + ViewModel + MPV view + observer + PiP + loaders + settings + utils + controls). It is self-contained, anime-only, and already decoupled from the manga reader except for the two utility imports noted above.
- **Keep MPV as the player library.** It is what aniyomi chose and what every anime source extension assumes (via `Video.mpvArgs`/`ffmpegStreamArgs`/`headers`). Switching to ExoPlayer/Media3 would require re-implementing the `Hoster`/`Video` → player-args translation for every source extension and would lose MPV-specific features (subtitle styling, custom Lua buttons, mpv.conf/input.conf user customization, gpu-next, debanding, AniSkip chapter merging). Do not switch.
- **Vendor the two manga-coupling points** into the player package or a shared `util/` package: move `SaveImageNotifier` from `ui/reader/` to `data/saver/` or `util/` (it has no reader dependencies), and either move the `isRecognizedNumber` extension or re-implement it locally. After that, the player package has zero manga imports.
- **Drop the manga reader entirely** for an anime-first build: `ui/reader/`, `presentation/reader/`, `ReaderPreferences`, `ReaderActivity`, `ReaderViewModel`, the manga `data/database/models/manga/` chapter model, `MangaDownloadManager`, manga domain interactors. The player does not need any of them.
- **Keep `:source-api`'s anime side intact** (`AnimeSource`, `AnimeHttpSource`, `ParsedAnimeHttpSource`, `Hoster`, `Video`, `Track`, `TimeStamp`, `ChapterType`). This is the contract every anime source extension implements.
- **Watch-outs:**
  - The `PlayerViewModel` constructor has ~20 Injekt-injected parameters — when forking, ensure all are still bound in `PreferenceModule`/`DomainModule`/`AppModule`.
  - `EpisodeLoader.checkHasHosters` uses reflection to detect new-vs-legacy source extensions. New extensions implement `getHosterList`/`hosterListRequest`/`hosterListParse`; legacy ones only implement `getVideoList(episode)`. Keep this dual-path until all bundled extensions migrate.
  - `Hoster.lazy = true` defers video-list fetching until the user expands that hoster in the `QualitySheet`. This is a deliberate bandwidth optimization for sources with many hosters — preserve it.
  - `video.initialized` gates whether `source.resolveVideo(video)` is called. Some sources return placeholder `Video`s with `videoUrl = "null"` that must be resolved before playback; `EpisodeLoader.getVideos` / `HosterLoader.getResolvedVideo` handle this. Do not skip the resolve step.
  - `MPV_ARGS_TAG = "ANIYOMI_MPV_ARGS"` metadata on downloaded videos carries per-video MPV options (set by the downloader from `Video.mpvArgs`). `PlayerActivity.setMpvOptions()` reads this metadata after `MPV_EVENT_FILE_LOADED` and applies each option. Preserve for downloaded playback.
  - The Lua bridge (`aniyomi.lua` + `custombuttons.lua`) is essential for custom buttons — if dropped, the entire custom-button feature (and the `CustomButton` domain model + `GetCustomButtons` interactor + editor screen) goes with it.
  - PiP `BroadcastReceiver` registration differs by SDK (`RECEIVER_NOT_EXPORTED` on Tiramisu+). Keep the branch.
  - `setupPlayerMPV()` writes `mpv.conf`/`input.conf` and copies scripts/shaders/fonts on every Activity create. This is intentional (user may have edited prefs) but is disk I/O on the main thread for the conf files (small) + a background coroutine for fonts. Acceptable but worth knowing.
  - `PlayerViewModel.onSecondReached` runs `saveEpisodeProgress` + `saveEpisodeHistory` every second via `launchNonCancellable`. That is 1 DB write/second per stream. The interactors batch into the SQLDelight `episodes`/`anime_history` tables. For low-end devices this could be a battery concern — consider throttling to every 5s if needed.
  - `runBlocking` is used in `initEpisodeList` (line 1260) and `checkTrackers` (line 1276) and `fileLoaded`'s AniSkip block (line 1194). These are inside `launchIO`/`launchNonCancellable` scopes but still block the coroutine — acceptable for now, but a potential jank source if DB/track calls are slow.
  - The `PlayerControls` Compose tree is large (654-line root + 5 corner controls + 7 sheets + 4 panels + 3 dialogs). It is rendered as a single `ComposeView` over the MPV `SurfaceView`. Performance is fine on modern devices but the recomposition scope is broad — watch for unnecessary recompositions when customizing.
  - Manifest `configChanges` includes `orientation|screenSize|uiMode` so the Activity is not recreated on rotation; `onConfigurationChanged` re-applies video aspect and hides controls in PiP. Preserve this.
