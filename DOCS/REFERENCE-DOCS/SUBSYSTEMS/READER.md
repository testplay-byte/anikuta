# Manga Reader

The aniyomi manga reader is a single-Activity (`ReaderActivity`) paged + webtoon image viewer that loads manga chapter pages from a `MangaSource` (HTTP, local archive, local directory, or downloaded `.cbz`), renders them with `SubsamplingScaleImageView` / Coil 3, and persists read progress + history to the manga SQLDelight DB. This is the manga-side reader; documented here for completeness — our build is anime-first and the reader is not required for anime playback.

## What it does

Opens a manga chapter selected from any manga screen, fetches its page list (from the source, the local-archive reader, or the download manager's disk cache), renders each page image in one of five reading modes, lets the user flip / scroll through pages with touch, volume keys, or D-pad, persists `last_page_read` and `read` flags per chapter, upserts a reading-history row with session duration, optionally pushes a tracker update, and enqueues adjacent-chapter downloads / post-read chapter deletion. The reader supports chapter preloading (prev + next), in-reader chapter transitions (gaps shown when source is missing chapters), dual-page spread splitting and rotation, color filters / grayscale / inversion, custom brightness, cutout-aware fullscreen, navigation tap-zone overlays, per-manga reading-mode and orientation overrides, and a page-action bottom sheet (save / share / copy / set-as-cover).

## Key files & classes

Activity + ViewModel:

- `ReaderActivity` (975 lines, `ui/reader/ReaderActivity.kt`) — extends `BaseActivity`. Inflates `reader_activity.xml`, observes `ReaderViewModel.state` and `eventFlow`, hosts the `Viewer`'s root `View` inside `binding.viewerContainer`, hosts the Compose overlay (`ReaderAppBars`, `ReaderContentOverlay`, all `Dialog`s) inside `binding.dialogRoot`, owns the `ReaderConfig` inner class for window-level config (background color, cutout, keep-screen-on, custom brightness, grayscale/invert layer paint, display profile). Entry: `newIntent(context, mangaId, chapterId)` puts two `Long` extras (`"manga"`, `"chapter"`) + `FLAG_ACTIVITY_CLEAR_TOP`. Dispatches `KeyEvent` / generic motion to the active `Viewer`.
- `ReaderViewModel` (996 lines, `ui/reader/ReaderViewModel.kt`) — `@JvmOverloads constructor` with 16 Injekt-injected dependencies (manga-only: `MangaSourceManager`, `MangaDownloadManager`, `MangaDownloadProvider`, `GetManga`, `GetChaptersByMangaId`, `GetNextChapters`, `UpsertMangaHistory`, `UpdateChapter`, `SetMangaViewerFlags`, `GetMangaIncognitoState`, `TrackChapter`; shared: `ReaderPreferences`, `BasePreferences`, `DownloadPreferences`, `TrackPreferences`, `LibraryPreferences`, `ImageSaver`, `SavedStateHandle`). Exposes `State` (immutable data class: `manga`, `viewerChapters`, `viewer`, `bookmarked`, `isLoadingAdjacentChapter`, `currentPage`, `dialog`, `menuVisible`, `brightnessOverlayValue`; derived `currentChapter` and `totalPages`) via `MutableStateFlow`, and `Event` (sealed interface: `ReloadViewerChapters`, `PageChanged`, `SetOrientation`, `SetCoverResult`, `SavedImage`, `ShareImage`, `CopyImage`) via `Channel.receiveAsFlow()`. Persists `chapter_id` + `page_index` in `SavedStateHandle` for process-kill restore.

Models (`model/`):

- `ReaderPage(index, url, imageUrl, stream)` — extends `Page` (from `:source-api`); adds `stream: (() -> InputStream)?` and a back-reference `chapter: ReaderChapter`. Subclassed by `InsertPage(parent)` for dual-page-spread halves.
- `ReaderChapter(chapter: Chapter)` — wraps the DB `Chapter` model; holds `state: State` (`Wait` / `Loading` / `Error(throwable)` / `Loaded(List<ReaderPage>)`) as a `MutableStateFlow`, plus `pageLoader: PageLoader?`, `requestedPage: Int`, and a refcount (`ref()` / `unref()` — when refcount hits 0 the page loader is recycled and state reset to `Wait`).
- `ViewerChapters(currChapter, prevChapter?, nextChapter?)` — triple of `ReaderChapter`s with `ref()` / `unref()` that propagates to each child.
- `InsertPage(parent: ReaderPage)` — copy of a `ReaderPage` representing the second half of a split wide image; `status = READY`, `stream = parent.stream`.
- `ChapterTransition` — sealed: `Prev(from, to?)` / `Next(from, to?)`. Carries the source and destination chapter for an in-reader transition card; equals/hashCode treat `Prev(A,B)` and `Next(B,A)` as the same transition.

Loaders (`loader/`):

- `PageLoader` (abstract) — `getPages(): List<ReaderPage>`, `loadPage(page)` (open, no-op default), `retryPage(page)` (open, no-op default), `recycle()` (sets `isRecycled`, `@CallSuper`), `isLocal: Boolean` (abstract).
- `ChapterLoader(context, downloadManager, downloadProvider, manga, source)` — single public method `loadChapter(chapter: ReaderChapter)`: returns immediately if already loaded; otherwise sets `State.Loading`, calls `getPageLoader(chapter)` (private) to pick the right `PageLoader`, assigns it to `chapter.pageLoader`, calls `loader.getPages()` (throws `page_list_empty_error` if empty), sets `chapter.requestedPage = chapter.chapter.last_page_read` (unless already read AND `preserveReadingPosition` is off), and sets `State.Loaded(pages)`. `getPageLoader` selection order:
  1. `downloadManager.isChapterDownloaded(..., skipCache = true)` → `DownloadPageLoader`.
  2. `source is LocalMangaSource` → dispatch on `source.getFormat(chapter)`: `Format.Directory` → `DirectoryPageLoader`, `Format.Archive` → `ArchivePageLoader(file.archiveReader(context))`, `Format.Epub` → `EpubPageLoader(file.epubReader(context))`.
  3. `source is HttpSource` → `HttpPageLoader(chapter, source)`.
  4. `source is StubMangaSource` → error (`source_not_installed`).
  5. else → error (`loader_not_implemented_error`).
- `HttpPageLoader(chapter, source, chapterCache = Injekt.get(), sourcePreferences = Injekt.get())` — `isLocal = false`. Uses a `PriorityBlockingQueue<PriorityPage>` consumed by a single coroutine on `Dispatchers.IO`; `preloadSize = 4`. `getPages()` reads `chapterCache.getPageListFromCache(...)` and falls back to `source.getPageList(chapter.chapter)` on cache miss; re-indexes pages with `ReaderPage(index, page.url, page.imageUrl)`. `loadPage(page)` re-enqueues ERROR/READY-but-evicted pages as priority 1 and preloads the next 4 pages at priority 0. `internalLoadPage(page)` fetches `source.getImageUrl(page)` if null, then `source.getImage(page, dataSaver)` and `chapterCache.putImageToCache(imageUrl, response)`, then sets `page.stream = { chapterCache.getImageFile(imageUrl).inputStream() }` and `page.status = READY`. `retryPage(page)` re-enqueues at priority 2. `recycle()` cancels scope, clears queue, and writes the page list back to `chapterCache` for faster reopen. `DataSaver` (SY) wraps `source.getImage` for bandwidth compression.
- `DownloadPageLoader(chapter, manga, source, downloadManager, downloadProvider)` — `isLocal = true`. `getPages()` calls `downloadProvider.findChapterDir(...)`; if it's a file (a `.cbz`/`.zip`/etc.), delegates to an inner `ArchivePageLoader(file.archiveReader(context))`; otherwise builds a page list from the download directory via `downloadManager.buildPageList(source, manga, chapter)` and returns `ReaderPage`s with `status = READY` and `stream = { context.contentResolver.openInputStream(page.uri) }`.
- `ArchivePageLoader(reader: ArchiveReader)` — `isLocal = true`. Uses `mihon.core.archive.ArchiveReader` (CORE-ARCHIVE); lists entries, filters to images via `ImageUtil.isImage`, sorts by case-insensitive natural order, and returns `ReaderPage(i)` with `stream = { reader.getInputStream(entry.name) }` and `status = READY`. `recycle()` closes the reader.
- `DirectoryPageLoader(file: UniFile)` — `isLocal = true`. Lists `file.listFiles()`, filters to images, sorts by natural order, returns `ReaderPage(i)` with `status = READY` and `stream = { file.openInputStream() }`. No `recycle()` override.
- `EpubPageLoader(reader: EpubReader)` — `isLocal = true`. Uses `mihon.core.archive.EpubReader`; calls `reader.getImagesFromPages()` and returns `ReaderPage(i)` with `stream = { reader.getInputStream(path) }` and `status = READY`. `recycle()` closes the reader.

Viewers (`viewer/`):

- `Viewer` (interface) — `getView(): View`, `destroy()` (default no-op), `setChapters(chapters: ViewerChapters)`, `moveToPage(page: ReaderPage)`, `handleKeyEvent(event: KeyEvent): Boolean`, `handleGenericMotionEvent(event: MotionEvent): Boolean`.
- `PagerViewer(activity)` (abstract) — wraps an AndroidX `ViewPager` via the custom `Pager` (extends `DirectionalViewPager`). Owns a `PagerConfig`, a `PagerViewerAdapter`, and the tap/long-tap routing. `onPageChange(position)` distinguishes `ReaderPage` vs `ChapterTransition` and forwards to `onReaderPageSelected` (which calls `activity.onPageSelected(page)`, requests preload of next chapter when within 5 pages of the end, and notifies the holder) or `onTransitionSelected` (requests preload of the destination chapter, or shows the menu if there is no next chapter). ## How it works

Open chapter (entry → ViewModel init):

1. A manga screen calls `startActivity(ReaderActivity.newIntent(context, mangaId, chapterId))`. Callers: `MangaScreen`, `MangaHistoryTab`, `MangaLibraryTab`, `MangaUpdatesTab`, `DeepLinkMangaScreen`, `NotificationReceiver` (new-chapter notification). All are manga-side; no anime screen starts `ReaderActivity`.
2. `ReaderActivity.onCreate` inflates `reader_activity.xml`, calls `viewModel.init(mangaId, chapterId)` if `viewModel.needsInit()` (i.e. `manga == null`). On `init`: `getManga.await(mangaId)`, wait for `MangaSourceManager.isInitialized`, set `state.manga`, look up `source = sourceManager.getOrStub(manga.source)`, construct `ChapterLoader(context, downloadManager, downloadProvider, manga, source)`, then `loadChapter(loader, chapterList.first { chapterId == it.chapter.id })`. `chapterList` is a `lazy` property built once: `getChaptersByMangaId.await(manga.id, applyScanlatorFilter = true)`, filtered by `skipRead`/`skipFiltered` (preserving the selected chapter even if filtered out), sorted by `getChapterSort(manga, sortDescending = false)`, de-duplicated (`skipDupe`), restricted to downloaded if `downloadedOnly`, mapped to `ReaderChapter`. SavedState restores `chapter_id` + `page_index`.
3. `ReaderActivity` observes `state.manga` (non-null) → `updateViewer()`: `ReadingMode.toViewer(viewModel.getMangaReadingMode(), this)` instantiates the viewer, destroys the previous viewer + removes its view, calls `viewModel.onViewerLoaded(newViewer)`, adds the new viewer's view to `binding.viewerContainer`, and creates a `ReaderProgressIndicator` in `binding.readerContainer`.
4. `ReaderActivity` observes `state.viewerChapters` (non-null) → `setChapters(viewerChapters)`: removes the loading indicator and calls `viewer.setChapters(viewerChapters)`; also fetches the chapter URL on a background thread for the assist / open-in-browser / share actions.

Load pages (ChapterLoader → PageLoader):

1. `ChapterLoader.loadChapter(chapter)` selects a `PageLoader` (see Key files above), calls `loader.getPages()`, attaches each returned `ReaderPage`'s `chapter` back-reference, sets `chapter.requestedPage` from `last_page_read` (or `0`), and transitions `chapter.state` to `Loaded(pages)`.
2. `HttpPageLoader.getPages()` prefers `chapterCache.getPageListFromCache(chapter)` and falls back to `source.getPageList(chapter.chapter)`. The page-list cache (`ChapterCache` in `data/cache/`) is a `DiskLruCache` at `context.cacheDir/chapter_disk_cache` keyed by chapter id; image entries are keyed by image URL.
3. `DownloadPageLoader` / `ArchivePageLoader` / `DirectoryPageLoader` / `EpubPageLoader` all return pages with `status = READY` and a `stream` lambda over a local file/content-provider/archive entry — no network round-trip.

Display in viewer (paged vs webtoon):

- Paged (`PagerViewer`): a `ViewPager` pages through the adapter's items. Each `ReaderPage` becomes a `PagerPageHolder` (a `ReaderPageImageView`); each `ChapterTransition` becomes a `PagerTransitionHolder`. The `ViewPager`'s `addOnPageChangeListener` invokes `onPageChange(position)` which routes to `onReaderPageSelected` or `onTransitionSelected`. R2L inverts the items list so the same adapter code works. Tap events are routed through `Pager.tapListener` → `config.navigator.getAction(pos)` → `toggleMenu` / `moveToNext` / `moveToPrevious` / `moveRight` / `moveLeft`. Wide-image split (`dualPageSplit`) emits an `InsertPage` via `PagerPageHolder.process` → `viewer.onPageSplit` → `adapter.onPageSplit` (inserts at `currentIndex + 1` for L2R/Vertical, `currentIndex` for R2L).
- Webtoon (`WebtoonViewer`): a `WebtoonRecyclerView` scrolls vertically through the same item shape. `WebtoonPageHolder.bind(page)` collects `page.statusFlow` and swaps progress indicator / image / error layout. Dual-page split uses `ImageUtil.splitAndMerge` (stacks halves vertically) rather than emitting a separate `InsertPage`. `CONTINUOUS_VERTICAL` is the same `WebtoonViewer` with `isContinuous = false` (adds a 15dp bottom margin between pages). Scroll distance per tap-zone activation = 3/4 screen height.

Image loading (Coil 3 + SubsamplingScaleImageView):

- `ReaderPageImageView` is the single rendering surface for both viewers. For non-animated images it uses `SubsamplingScaleImageView` (paged) or `WebtoonSubsamplingImageView` (webtoon); for animated images it uses `PhotoView` (paged) or `AppCompatImageView` (webtoon) driven by a Coil 3 `ImageRequest`. Coil 3 (`io.coil-kt.coil3:coil-bom:3.1.0` per `gradle/libs.versions.toml`) is also used as a fallback decoder for webtoon non-animated images when `alwaysDecodeLongStripWithSSIV` is off; the request disables memory and disk cache and uses the `cropBorders()` and `customDecoder(true)` Coil mappers from `data/coil/`. The custom `TachiyomiImageDecoder` and `SubsamplingScaleImageView.setDisplayProfile(...)` honor the user's display-profile ICC file. Max zoom = 5x, double-tap zoom = 2x with system-scaled duration.

Save progress (onPageSelected → DB):

1. `Viewer.onPageSelected` → `ReaderActivity.onPageSelected(page)` → `ReaderViewModel.onPageSelected(page)`.
2. `onPageSelected` ignores `InsertPage`s, launches `updateChapterProgress(selectedChapter, page)` non-cancellable, and if `selectedChapter != getCurrentChapter()` calls `loadNewChapter(selectedChapter)` to swap the active chapter. If `page.number / pages.size > 0.25` and `autoDownloadWhileReading > 0`, calls `downloadNextChapters`.
3. `updateChapterProgress` sets `state.currentPage = pageIndex + 1`, `readerChapter.requestedPage = pageIndex`, persists `chapterPageIndex` to `SavedStateHandle`. If not incognito and the page is not in ERROR state: sets `readerChapter.chapter.last_page_read = pageIndex`; if at the last index, calls `updateChapterProgressOnComplete` (sets `read = true`, fires `updateTrackChapterRead`, calls `deleteChapterIfNeeded`, and — if `markDuplicateReadChapterReadAsRead` includes `MARK_DUPLICATE_CHAPTER_READ_EXISTING` — bulk-marks duplicate-numbered chapters as read via `updateChapter.awaitAll`); then `updateChapter.await(ChapterUpdate(id, read, lastPageRead))` writes to the `chapters` table.
4. `flushReadTimer` (on `onPause` / `onActivityFinish`) → `updateHistory(readerChapter)` → `upsertHistory.await(MangaHistoryUpdate(chapterId, readAt, sessionReadDuration))` writes a row to the `history` table (manga DB). `chapterReadStartTime` is reset by `restartReadTimer` (on `onResume` and on chapter change).
5. `updateTrackChapterRead` skips if incognito or `trackPreferences.autoUpdateTrack()` is off; otherwise `trackChapter.await(context, manga.id, readerChapter.chapter.chapter_number.toDouble())` updates all manga trackers (see TRACKERS.md).
6. `deleteChapterIfNeeded` reads `downloadPreferences.removeAfterReadSlots()`; if non-negative (-1 = never), finds the chapter `removeAfterReadSlots` positions before the current one and `enqueueDeleteReadChapters` via `downloadManager.enqueueChaptersToDelete(...)` (deferred until `onActivityFinish` → `deletePendingChapters`).
7. `downloadNextChapters` only fires if the current chapter's loader is a `DownloadPageLoader` (i.e. the current chapter is fully downloaded), checks the next chapter is also downloaded, then `getNextChapters.await(manga.id, nextChapter.id)` (optionally de-duped) `.take(downloadAheadAmount)` → `downloadManager.downloadChapters(manga, chaptersToDownload)`.

Reading direction (LTR / RTL / vertical / webtoon):

- Stored on the `Manga` entry as a 3-bit `readingMode` flag (masked `0x7`); resolved via `ReaderViewModel.getMangaReadingMode(resolveDefault = true)` which falls back to `readerPreferences.defaultReadingMode().get()` (default `RIGHT_TO_LEFT`, i.e. manga-style) when the per-manga flag is `DEFAULT`. `ReadingMode.toViewer(...)` instantiates the matching `Viewer` (see Key files). `setMangaReadingMode(mode)` calls `setMangaViewerFlags.awaitSetReadingMode(manga.id, mode.flagValue.toLong())` then reloads `viewerChapters` (without re-loading pages) and emits `ReloadViewerChapters`. Orientation (`readerOrientation` 3-bit flag, masked `0x38`) is applied via `Activity.requestedOrientation` and persisted similarly.

Transitions:

- `ChapterTransition.Prev` / `Next` cards are inserted between chapters by both adapters. A transition is forced when `calculateChapterGap` (delegates to `tachiyomi.domain.items.chapter.service.calculateChapterGap`) returns > 0 (missing chapters) or when the prev/next chapter isn't yet `Loaded`, or when `alwaysShowChapterTransition` is on. `PagerTransitionHolder` / `WebtoonTransitionHolder` host a `ReaderTransitionView` (Compose `ChapterTransition` composable) that shows the from/to chapter names and download status, plus a loading spinner or error+retry button observing `transition.to.stateFlow`. Reaching a transition triggers `activity.requestPreloadChapter(to)` (or `activity.showMenu()` if there is no next chapter).

Settings:

- Window-level: `ReaderActivity.ReaderConfig` inner class subscribes to `readerTheme`, `displayProfile`, `cutoutShort`, `keepScreenOn`, `customBrightness` (+ value, sampled at 100ms), `grayscale` + `invertedColors` (combined into a `ColorMatrixColorFilter` applied via `setLayerType(LAYER_TYPE_HARDWARE, paint)`), and `fullscreen` (toggles `WindowCompat.setDecorFitsSystemWindows` and re-applies viewer insets). Background color: 0=white, 1=black (default), 2=gray (`#202125`), 3=automatic (night-mode aware). Brightness overlay (semi-transparent black, value range -75..100) is rendered by the Compose `ReaderContentOverlay` driven by `state.brightnessOverlayValue`.
- Viewer-level: `PagerConfig` / `WebtoonConfig` subscribe to image-scale, zoom, crop, navigation, dual-page prefs; their `imagePropertyChangedListener` triggers `refreshAdapter` which recreates the visible holders. `navigationModeChangedListener` updates the `ReaderNavigationOverlayView`.
- Color filter overlay: `ReaderContentOverlay` Composable blends `colorFilterValue` with the chosen `BlendMode` (SrcOver / Modulate / Screen / Overlay / Lighten / Darken) over the whole reader.
- All settings UI lives in `presentation/reader/settings/` (`ReaderSettingsDialog` as the host, `GeneralSettingsPage`, `ReadingModePage`, `ColorFilterPage`) and is shown via `state.dialog = Dialog.Settings`. Mode/orientation selection uses dedicated `ReadingModeSelectDialog` / `OrientationSelectDialog`. Page actions use `ReaderPageActionsDialog` (save / share / copy / set-as-cover).

## Anime vs manga

The reader is **fully manga-specific**. Every dependency is manga-namespaced: `MangaSourceManager`, `MangaDownloadManager`, `MangaDownloadProvider`, manga domain interactors (`GetManga`, `GetChaptersByMangaId`, `GetNextChapters`, `UpsertMangaHistory`, `UpdateChapter`, `SetMangaViewerFlags`, `GetMangaIncognitoState`, `TrackChapter`), manga DB models (`eu.kanade.tachiyomi.data.database.models.manga.*`), manga source API (`eu.kanade.tachiyomi.source.online.HttpSource`, `eu.kanade.tachiyomi.source.MangaSource`, `tachiyomi.source.local.entries.manga.LocalMangaSource`), manga source-local `Format`/`isLocal`. There are zero anime-side imports anywhere in `ui/reader/`, `presentation/reader/`, or `loader/`/`model/`/`viewer/`/`setting/` — verified by `grep "ui.player" ui/reader/` returning no matches.

Relationship to the PLAYER (anime):

- **Separate Activities**: `ReaderActivity` (manga) vs `PlayerActivity` (anime). Both are `singleTask` (manifest lines 184 and 205). No shared base Activity class (both extend `BaseActivity` only).
- **Separate ViewModels**: `ReaderViewModel` (996 lines) vs `PlayerViewModel` (~2059 lines). No shared base ViewModel; no shared state holders.
- **Separate viewer stacks**: `ReaderPageImageView` / `SubsamplingScaleImageView` / `PhotoView` / `ViewPager` / `RecyclerView` (reader) vs `AniyomiMPVView` / `is.xyz.mpv.MPVLib` JNI bindings (player). No shared rendering code.
- **Separate settings**: `ReaderPreferences` + `presentation/reader/settings/*` vs `PlayerPreferences` + `presentation/more/settings/screen/player/*`. Shared only via the `presentation/components/` primitives (AppBar, sheets, dialogs) used app-wide.
- **Separate progress tables**: `chapters` + `history` (manga DB) vs `episodes` + `anime_history` (anime DB). The reader writes only to the manga DB; the player writes only to the anime DB.
- **Cross-check vs PLAYER.md**: PLAYER.md "Anime vs manga" lists two trivial manga-coupling points in the player: (1) `PlayerViewModel` line 80 imports `SaveImageNotifier` from `ui.reader` (verified — line 1734 uses it); (2) `PlayerViewModel` line 61 imports `data.database.models.manga.isRecognizedNumber` (verified — used by the duplicate-numbered-episodes feature). PLAYER.md notes both could be moved to a shared `util/` package to fully decouple the player from the reader. The reader itself has no reciprocal dependency on the player.

The directionality is one-way: **the player depends on two utilities that live in the reader's package; the reader does not depend on the player at all**. Removing the reader for an anime-first build requires relocating (or re-implementing) `SaveImageNotifier` and the manga `isRecognizedNumber` extension so the player continues to compile.

## Notes for our build (anime-first)

The reader is **not needed** for an anime-first build. It can be deleted in full without affecting the player. Concretely:

- **What to drop**: `app/src/main/java/eu/kanade/tachiyomi/ui/reader/` (entire package — Activity, ViewModel, loader/, model/, viewer/, setting/, `SaveImageNotifier`, `ReaderNavigationOverlayView`), `app/src/main/java/eu/kanade/presentation/reader/` (entire Compose UI), `app/src/main/res/layout/reader_activity.xml` + `reader_error.xml`, the `<activity android:name=".ui.reader.ReaderActivity">` block in `AndroidManifest.xml`, the `ReaderPreferences` registration in `PreferenceModule`, the `ChapterCache` binding in `AppModule` (only the reader consumes it), the manga-side caller sites in `MangaScreen` / `MangaHistoryTab` / `MangaLibraryTab` / `MangaUpdatesTab` / `DeepLinkMangaScreen` / `NotificationReceiver` (these would already be removed when the manga UI is dropped), and the manga `<string>` resources referenced only by reader strings.
- **What to relocate before deleting**: (1) `SaveImageNotifier` — move from `ui/reader/` to `data/saver/` or `util/notification/` so `PlayerViewModel` line 80 import + line 1734 usage still compile. It has no reader dependencies (only Coil 3 + `NotificationHandler` + `NotificationReceiver`). (2) `isRecognizedNumber` extension on manga `Chapter` — either re-implement locally on anime `Episode` (the player already imports the anime variant on line 59; the manga import on line 61 may be dead code or used for a manga-only branch — TODO to verify) or move it to a shared `data/database/models/util/` package.
- **Cost of re-adding later**: if manga support is added back, the reader can be restored near-verbatim from the reference. The integration points are mechanical: re-register `ReaderPreferences` in `PreferenceModule`, re-register `ChapterCache` in `AppModule`, re-add the `<activity>` to the manifest, re-wire the manga-side `ReaderActivity.newIntent` callers (which presupposes the manga UI screens exist), and ensure the manga DI graph (`MangaSourceManager`, `MangaDownloadManager`, `MangaDownloadProvider`, manga domain interactors, manga SQLDelight DB) is present. The reader has no evolutionary coupling to the player — they were never coupled to begin with — so re-adding the reader has no player-side prerequisites.
- **Trade-off note**: dropping the reader removes ~5k LOC of UI/viewer/loader code plus the `ChapterCache` infrastructure, but leaves the player with a single ~30-line utility dependency (`SaveImageNotifier`) that must be relocated. The trade-off is asymmetric: the player is self-contained after the relocation; the reader is self-contained always (it has no player dependency). For an anime-first build, dropping the reader is a clear win — the only ongoing cost is keeping `SaveImageNotifier` in a shared location if both packages coexist again later.
